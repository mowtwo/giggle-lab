# 《赵云与阿斗》联机对战 —— 架构与协议规范 v0.2

> 目标:**完整实时 1v1 对战,真正实时 + 公平**,且支持**跨边副作用技能**(一方技能影响对方
> 单位)。客户端**静态部署**,服务器由玩家**自建**、尽量只做**哑中继**(转发 + 房间),
> 不跑游戏逻辑。客户端配置服务器地址即可。
>
> 本文是客户端与**任意兼容服务器**之间的契约。服务器单独成仓库,照此协议实现即可互通。

---

## 0. 为什么是 lockstep(确定性) —— 公平性论证

候选架构与"公平 + 实时 + 跨边技能"的匹配:

| 架构 | 公平? | 实时? | 跨边技能 | 服务器 | 结论 |
|---|---|---|---|---|---|
| 主机权威(host 跑双方,下发快照) | ✗ host 0 延迟 / guest 1×RTT | ✓ | host 结算 | 哑中继 | **不公平,否决** |
| 服务器权威(服务器跑模拟) | ✓ 双方对称 | ✓(需预测) | 服务器结算 | **要跑游戏逻辑** | 公平,但服务器不再"哑" |
| **确定性 lockstep(推荐)** | ✓ 双方对称输入延迟 | ✓ | 两端按同一输入流确定性结算 | **哑中继(只转发输入)** | **满足全部约束** |

**lockstep 原理**:两端运行**完全一致的确定性模拟**,网络上只传"玩家操作输入";所有输入
都**延迟同样的 N 个逻辑帧**(input delay)后在两端同时生效。延迟对称 ⇒ 公平;同一份有序
输入流 + 确定性模拟 ⇒ 两端逐帧一致,跨边副作用技能天然一致;服务器只转发输入 ⇒ 仍是哑中继。

**前置条件(本项目最大工作量)**:模拟必须**绝对确定**。当前引擎不是——见 §2。所以要先把
战斗重做成**确定性核心**。该核心一旦完成,lockstep 与"服务器权威"都能复用它(网络权威可后选)。

## 1. 总体架构

```
        ┌───────── 哑中继服务器(自建,只转发输入/房间) ─────────┐
        └──────────────┬───────────────────────┬──────────────┘
                       │ WS(输入流)            │ WS(输入流)
              ┌────────▼─────────┐    ┌─────────▼────────┐
              │  客户端 A         │    │  客户端 B          │
              │ 确定性核心(逻辑) │ == │ 确定性核心(逻辑)  │  ← 两端逐帧一致
              │ Laya 渲染(表现)  │    │ Laya 渲染(表现)   │
              └──────────────────┘    └──────────────────┘
```

- **确定性核心(Sim)**:纯逻辑、定步长(如 30Hz)、种子随机、无 `Date.now`、无 tween 驱动状态。
  吃"输入帧",吐"世界状态"。两端给定相同 `seed + 输入序列` ⇒ **完全相同的状态**。
- **渲染层(View)**:Laya 把 Sim 的世界状态画出来;tween/动画**只做插值与特效**,**不回写**任何
  影响判定的状态(Bw/mL/hp/位置都由 Sim 决定)。
- **网络**:每个逻辑帧两端交换各自的输入(可空),延迟 N 帧生效;服务器原样转发。

## 2. 确定性核心改造清单(M1,最大块)

把现有战斗从"实时 + tween 驱动"改造为"定步长 + 状态驱动":

1. **种子随机**:`MathE` 的 `Math.random` 换成可注入的种子 PRNG(mulberry32/xorshift32)。
   全局 ~43 处随机绝大多数走 `MathE.range/weightedIndex/sample`,改一处即覆盖;少量直接
   `Math.random` 收编进 `MathE`。开局用协议下发的 `seed` 初始化。
2. **定步长逻辑**:战斗逻辑从"按真实 `delta`"改为"按固定 `dtFixed`(如 33ms)"推进。
   `UpdateMgr` 渲染帧累积时间,凑够一个 `dtFixed` 跑一次 Sim step(可一帧跑多步追赶)。
3. **tween 与状态解耦(关键、量最大)**:目前 `Bw`(可被瞄准)、`mL`(可行动)由
   Laya.Tween 的 `.then()` / 动画 `STOPPED` 事件按真实时间翻转 —— 必须改为**由 Sim 用逻辑帧
   计时**(如"冻结剩余帧数"倒计时到 0 才恢复 `Bw`)。涉及:enemy(出场/冻结/击飞)、
   general/soldier(攻击动画门控 mL)、weapons、skills、buff、props。tween 退化为纯表现。
   *(上一轮加的 Bw/mL 看门狗在确定性核心里会被"逻辑帧计时"取代,不再需要兜底。)*
4. **确定的迭代顺序**:实体遍历按稳定顺序(现用 `Map`,JS 按插入序迭代,基本 OK;需审计
   有无依赖对象地址/浮点误差累积的地方)。
5. **浮点一致性**:同为 JS/同引擎,双端浮点一致;避免 `Math` 中平台相关函数即可。坐标可
   考虑定点化(后续优化,不是 v0.2 必须)。
6. **可离线自检**:Sim 必须能脱离 Laya 跑(headless),用于 §9 的"同种子同输入 ⇒ 同状态"
   单测与回放。

> 验收:给两个 Sim 实例相同 `seed` 与相同输入序列,跑 N 帧后**世界状态哈希逐帧相等**。
> 这是公平实时联机的硬门槛,M1 必须通过。

## 3. 传输与编码

- **传输**:WebSocket(`ws`/`wss`)。房间发现/匹配用同源 HTTP(§6)。
- **编码**:JSON(v0.2);输入帧极小(每帧每人 0~数个操作),带宽不是瓶颈。
- **信封**:`{ "t":type, "v":2, "frame":N, "data":{...} }`(`v`=协议版本=2)。

## 4. lockstep 同步细节(难点 #3)

- **逻辑帧编号** `frame`:从开局 0 递增,每 `dtFixed` 一帧。
- **输入延迟** `INPUT_DELAY`(如 3 帧 ≈ 100ms):玩家在帧 F 产生的操作,标记为在帧
  `F + INPUT_DELAY` 生效,发给对端;两端都在该帧施加 ⇒ **双方延迟对称、公平**。
- **帧推进闸门**:Sim 只有在**已收齐"该帧两端的输入"**时才推进该帧(空输入也要发
  `INPUT { frame, ops:[] }` 作为心跳)。收不齐 ⇒ 暂停等待(网络卡顿表现为双方一起轻微卡顿,
  而非一方领先 ⇒ 仍公平)。
- **跨边副作用技能**:就是普通输入(`OP prop`),两端在同一生效帧、按确定性核心同时结算,
  对对方单位的影响逐帧一致,无需特殊处理。
- **延迟自适应**:测 RTT 动态调 `INPUT_DELAY`(2~6 帧),平衡手感与卡顿。
- **断线**:超时收不到对端 `INPUT` ⇒ 暂停 N 秒;恢复则带 `frame` 续跑,超时判离开方认输。
- **防作弊(自建友好局,弱要求)**:可选每 K 帧交换"状态哈希",不一致即判定 desync 并中止
  (用于发现确定性 bug,而非对抗恶意改包)。

## 5. 玩家身份(难点 #4)

- `SaveMgr` 持久化:`playerId`(本地生成 UUID,自声明身份)、`playerName`(可编辑)。
- 房间/匹配/输入消息携带 `{ playerId, playerName }`。战斗 UI 用对手 `playerName` 替换"AI"。
- 服务器不做账号系统;`playerId` 仅作连接/重连标识。

## 6. 房间与匹配(HTTP,同源服务器地址)

客户端配置 `serverBaseUrl`。端点:

| 方法 | 路径 | 作用 | 响应 |
|---|---|---|---|
| GET | `/adou/health` | 探活 + 协议版本 | `{ ok, protocol:2, name }` |
| GET | `/adou/rooms` | 公开房间列表 | `{ rooms:[{id,name,hostName,state,players}] }` |
| POST | `/adou/rooms` | 创建房间 `{playerId,playerName,mapId,private}` | `{ roomId, wsUrl }` |
| POST | `/adou/rooms/{id}/join` | 加入 `{playerId,playerName}` | `{ roomId, wsUrl }` |

`state`: `waiting|ready|playing|finished`。`wsUrl` 可带一次性 ticket。

---

## 7. WebSocket 消息

| `t` | 方向 | data | 说明 |
|---|---|---|---|
| `HELLO` | C→S | `{playerId,playerName,roomId,ticket}` | 入房握手 |
| `ROOM_STATE` | S→C | `{players:[{playerId,name}],state}` | 在场变化广播 |
| `READY` | C→S | `{ready}` | 准备 |
| `MATCH_START` | S→C | `{seed, mapId, startAtMs, players:[{playerId,name,side}], inputDelay}` | 双方就绪后开局,下发种子 |
| `INPUT` | C↔S | `{frame, ops:[Op...]}` | **每逻辑帧必发**(含空帧心跳);服务器转发给对端 |
| `HASH` | C↔S | `{frame, hash}` | 可选,周期性状态哈希校验 desync |
| `PEER_LOST` | S→C | `{playerId}` | 对端掉线 |
| `MATCH_END` | C↔S | `{winnerId, reason, atFrame}` | 任一端裁定并广播(自建局信任) |

`Op`(操作输入):
```jsonc
{ "op":"place|merge|move|prop|recruit|sell",
  "unit":"...", "from":{c,x,y}, "to":{c,x,y}, "propId":12, "target":{c,x,y} }
```
> 注意:lockstep 下**没有"快照下行"** —— 两端各自用确定性核心算出全部单位,网络只跑输入。

## 8. 客户端落地(M2/M3,核心完成后)

1. `NetMgr`(新单例):连接/房间/收发,`mode = local | net`;传输可换(真 WS / loopback)。
2. `SaveMgr`:`playerId` / `playerName`。
3. 战斗接入:
   - 单机:本地输入直接进 Sim(`INPUT_DELAY=0`),沿用现 AI 作对手。
   - 联机:本地操作 → `Op` 入本地输入队列(延迟 N 帧)+ 发给对端;每帧收齐两端输入再推进 Sim;
     **AICtr 在联机模式停用**,对手侧由"对端输入"驱动同一个确定性核心。
4. 主菜单:服务器地址配置、房间列表/创建/加入、玩家名编辑。
5. `adou-local-bootstrap.js`(单机假后端)保持不变,仅服务于排行等非对战上报。

## 9. 测试策略(难点 #1)

- **L0 确定性自检(最重要)**:headless 跑两个 Sim,同 `seed` + 同输入序列,**逐帧状态哈希
  必须相等**;随机化输入做模糊测试。这是 M1 的硬验收,也是防 desync 的根本保障。
- **L1 页面内环回**:`NetMgr` 的 `loopback` 传输,同页两个对局实例互发 `INPUT`,**无需服务器**
  即可验证 lockstep 闸门、输入延迟、暂停/恢复。
- **L2 本地真服务器**:跑参考中继于 `localhost`,两标签页/两同网设备各配 `ws://<ip>:PORT`。
- **L3 弱网**:传输层注入延迟/抖动/丢包,验证帧闸门与重连;开 `HASH` 校验抓 desync。

## 10. 服务器仓库(难点 #2)

单独仓库,**只实现 §6 + §7 的房间与输入转发**,无游戏逻辑(lockstep 下服务器极简):

- 技术栈:Node + `ws`(或 Go);内存房间表起步。
- 职责:HTTP 房间 CRUD + 探活;WS 绑定房间;**把 `INPUT`/`HASH`/控制消息原样转发对端**;
  在场/掉线广播;可选公开房间列表/密码/重连 ticket。
- 交付:本协议文档(契约)+ 可 `npm start` 的参考中继 + Dockerfile。玩家自建后在游戏里填
  `serverBaseUrl` 即可。

## 11. 里程碑(核心优先)

- **M0**:协议文档(本文件)。
- **M1(最大块)**:确定性核心改造 —— 种子随机 + 定步长 + tween/状态解耦 + headless;
  通过"同种子同输入 ⇒ 逐帧状态哈希相等"自检。**这是公平实时联机的前置硬门槛。**
- **M2**:`NetMgr` + loopback + lockstep 闸门/输入延迟;同页双实例跑完整一局且两端一致。
- **M3**:服务器仓库 + 参考中继;两标签页/两设备真实联机;id/名字/房间 UI。
- **M4**:延迟自适应、断线重连、`HASH` 防 desync、弱网手感、(可选)二进制/定点优化。
